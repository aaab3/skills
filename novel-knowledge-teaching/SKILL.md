# 小说讲知识：叙事性知识传授方法论

> 本技能源自一份完整的 LLM 科普小说创作设计（用户378525提供），
> 经分析提炼为通用方法论，可应用于任何技术概念的叙事化讲授。

---

## 核心洞察

**形式即内容。**

用 in-context learning 的方式讲 in-context learning：
- 不直接告诉答案，而是给例子，让读者自己悟
- 信号系统 = 给读者建 watch list，不是替读者懂
- 隐喻崩塌附录 = 对懂行的诚实，也是对读者的尊重

---

## 一、为什么"用小说讲技术"通常失败

两种经典失败品：

1. **教科书穿戏服** — 人物只是讲师的传声筒，章末附"知识点小结"
2. **小说里夹百科** — 故事归故事，技术归技术，读者跳着读

**破局关键：** 不是"在小说里讲技术"，而是让技术概念本身变成人物、地理、情节。

---

## 二、叙事结构的顶层设计：三线交织

| 线 | 视角 | 功能 | 语言特征 |
|---|------|------|---------|
| **外篇·人间** | 研究员/工程师 | 现实主线：训练/伦理/产品 | 现实主义时态，细腻心理描写 |
| **内篇·模型内部** | token"词" | 寓言主线：每次推理=穿越之旅 | 诗意/短句/意象密集 |
| **古篇·谱系** | 历史短札 | 思想史：先驱未解之题 | 文献体/冷峻/片札式 |

**三线汇合：** 外篇人物问出的问题 = 内篇里"词"在塔顶遇到的谜 = 古篇某先驱未解的题。

**好处：** 任意层次读者都能进入。文学读者读外篇即可，技术读者看内篇精确隐喻，思想读者被古篇钩住。

---

## 三、核心设计原则（五条铁律）

### 1. 角色即概念
不要"介绍"注意力机制——让 Q（Query）角色与 K、V 在每一章舞台上重复跳舞，读者自然看懂。

### 2. 冲突驱动揭示
每个技术点必须由人物的某个痛逼出来。
- 模型胡说八道 → 引出"幻觉"的解释
- 模型给富人和穷人不同建议 → 引出 RLHF 的偏置

### 3. 单一统御性隐喻
整个作品只允许**一个**核心隐喻体系。所有概念在它里面自洽。混用隐喻是大忌。

**示例：字之城**
- 街道 = embedding 空间
- 议会堂 = attention 机制
- 城墙 = context window
- 铸币厂 = tokenizer

### 4. 留白胜过定义
让读者自己拼出"哦，原来 embedding 就是这个"，胜过角色一句"所谓 embedding 就是…"。

这本身和 LLM 的 in-context learning 同构——很美。

### 5. 附录补严谨
正文重诗意，书末附"隐喻在何处崩塌"一章，老实说每个比喻在哪里失真。这是对科学的诚实，也是对读者的尊重。

## 四、三层信号系统（关键创新）

**用户的核心洞察：** 直接加注释有三个坑——戳破魔法、变成教科书、打断节奏。

**正确做法不是注释，是"信号"：让读者带着地图读小说。**

---

### 第一层：章末"暗格"（必有）

每章结尾留一个加框小段。不是注释，不是总结，而是把这一章最重要的隐喻点一下名——**但不解释**。

```
┌─────────────────────────────────┐
│ ◐ 第一章 · 暗格                   │
│                                   │
│  这一章你看见的：                   │
│  · 一把刀，把字切成半截             │
│  · 一条永远在的主干道               │
│  · 一个不说话只点头的老者           │
│  · 一座九十六层、最高一层是黑的塔    │
│                                   │
│  把它们记一下。后面会回来。         │
└─────────────────────────────────┘
```

**纪律：**
- 不写"这是什么的隐喻"
- 不写技术词
- 只列"你看见的具体形象"
- 加一句"后面会回来"

---

### 第二层：极少数关键句的"侧批"（节制使用）

整部作品最多用 **30-40 处**。位置在段落后或右侧栏。

**样式：**
> 它低头看自己。它现在的样子是 ##ization。前面那两个井号像两道疤。它不知道井号是什么意思。它只知道——从此以后，它不能独自站着了。它必须粘在别人后面，永远做个尾巴。
>
> ⟡ 它说"两道疤"。记住这个词。

**纪律：**
- 永远是指认，不是解释
- 用"记住""注意""它说的是""不是偶然"这种词
- 绝不出现技术名词
- 像京剧的板鼓——只在该响的地方响一下

---

### 第三层：章末"地图条"（最重要）

每章正文之后，翻一页，是一张极简卡片。

```
 ◐ 第一章 · 你刚刚学会了什么

 半截 = 词被切碎
 主干道 = 一条不参与决策、只负责保留原貌的通道
 老者 = 主干道的化身，他记下你说的每一句
 九十六层塔 = 你会一层一层往上走，每上一层，世界就被重写一次

 你不需要现在懂这些。
 你只需要在它们再次出现的时候，认出它们。
```

**为什么这是对的：**
- 不在正文里，所以不破坏沉浸
- 翻一页才看见，给读者选择权
- 用寓言语言，不是技术语言
- "你不需要现在懂"这句话本身就在降低焦虑——这是把读者留下来的关键

---

### 第四层：书末"全图揭示"（可选）

读者读完最后一章，翻到附录，看到：

```
你一路看见的那些东西，它们的真名是——

半截      = tokenization
主干道    = residual connection
老者      = residual stream
九十六层塔 = Transformer 的 96 层
……
```

这是"啊哈时刻"。读者会想立刻翻回去重读一遍。

---

### 信号系统效果对比

| 方法 | 当下沉浸感 | 知识留存 | 重读价值 |
|------|-----------|---------|---------|
| 不加任何注释 | 满分 | 低 | 中 |
| 直接加技术注释 | 大破坏 | 中 | 低 |
| 三层信号系统 | 几乎不影响 | 高 | 极高 |

**核心差别：注释是替读者懂（一次性消耗），信号是请读者自己懂（持续生效）。**

## 五、章节-概念映射方法论

### 映射表模板

每章需要对齐以下五栏：

| 章节 | 内篇情节 | 外篇情节 | 暗藏技术 | 古篇片札 | 哲学母题 |
|------|---------|---------|---------|---------|---------|
| 示例：第3章《议会》 | 词进议会堂举两块牌 | 调长上下文出bug | Self-Attention | 注意力是稀缺资源（Britannica） | 谁有权被听见 |

### 验收标准

**每章必须同时满足：**
1. 抽掉技术，故事依然成立
2. 不读附录，概念能感受到
3. 这章有情绪（没有情绪的章节一定要重写）

## 六、通用写作流程

### 第一步：确立核心隐喻
在开始写之前，必须锁定**唯一**的统御性隐喻。这是整部作品的"物理学"。

### 第二步：建立概念-意象映射表

```
概念               →  隐喻意象
─────────────────────────────────
tokenization       →  刀切字/半截
embedding空间     →  街道/公寓/城区
self-attention    →  议会堂/举牌喊话
transformer层     →  九十六层塔
residual connection→  主干道
RLHF             →  老师的标尺
emergence        →  塔顶亮灯
……
```

### 第三步：写外篇人物弧光
先把外篇人物写完整——人物为技术受苦，不是为技术受苦。

### 第四步：按三线结构编织
外篇/内篇/古篇不是交替出现的段落，而是**同一时刻不同维度的投影**。

### 第五步：章末三检

每章完成后自查：
- [ ] 抽掉技术，故事还成立吗？
- [ ] 不读附录，概念能感受到吗？
- [ ] 这章的情绪是什么？

## 七、为任何技术领域创建叙事化讲授的步骤

本方法论不限于 LLM，可应用于任何复杂概念的科普：

### Step 1：找核心隐喻
问：这个领域最直觉的意象是什么？
- 网络安全 → 城堡/护城河/间谍
- 量子计算 → 平行世界/叠加态/观测坍缩
- 生物进化 → 树/分支/适者生存

### Step 2：设计角色体系
问：哪些角色是这个世界的居民？
- 他们各自代表什么概念？
- 他们之间的关系是什么（冲突/依赖/等级）？

### Step 3：构建三线结构
问：同一现象的三个叙事维度是什么？

### Step 4：制作章节-概念映射表
每章对应一个核心概念，按叙事顺序排列。

### Step 5：植入三层信号
在写作过程中，按三层信号系统的规范在适当位置添加信号。

### Step 6：撰写附录
书末写"隐喻在何处崩塌"——主动认错，对懂行的尊重。

## 九、快速启动模板

当用户说"帮我用小说讲[X概念]"时：

1. 询问核心受众是谁（技术人/普通人/混合）
2. 确定是否已有核心隐喻，或需要新建
3. 如果需要新建：
   - 请用户描述这个概念最直觉的意象
   - 列出 3-5 个最难讲清楚的概念点
4. 输出一份 7-12 章的叙事大纲（参考 21 章 LLM 示例）
5. 开始写第一章样章

## 十、常见失败模式

| 失败模式 | 特征 | 解决方法 |
|---------|------|---------|
| 教科书穿戏服 | 人物大段讲解技术 | 删掉讲解，改成人物因痛而说 |
| 隐喻打架 | 一会儿用城市，一会儿用战场 | 锁死唯一统御隐喻 |
| 信号变注释 | 侧批出现技术名词 | 绝对禁止技术词，只许指认 |
| 没有情绪 | 技术推进顺利，人物无反应 | 先写外篇人物弧光 |
| 古篇太学术 | 大段引用无铺垫 | 古篇永远是片札，从不是正文 |
| 观点库评论体 | 金句对仗、无个人场景 | 每段加可验证细节（见第十一节） |
| 误用爽文节拍 | 章章反转、系统面板、密集打脸 | 知识长篇用「认知渐显」；见 `references/contemporary-structures-and-markets.md` |
| 外篇断线 | 连续多章只有内篇寓言 | 外篇每 2–3 章至少一场具体人间戏 |

## 十一、叙事运输：为什么信号系统有效（心理学底座）

**叙事运输（Narrative Transportation）**（Green & Brock）：读者高度沉浸故事时，更少反驳、更易形成故事一致信念、记忆更好。见 [Transportation theory](https://en.wikipedia.org/wiki/Transportation_theory_(psychology))。

**设计推论（写技能时必须遵守）：**

| 做法 | 对运输的影响 |
|------|----------------|
| 正文直接写技术定义 | 打断运输 → 变上课 |
| 章末「暗格」只列形象 | 建 watch list，不戳破魔法 |
| 外篇私人场景（录音、删不删数据） | 拉高情感运输 |
| 书末「隐喻崩塌」表 | 运输结束后的二次啊哈，促重读 |
| 古篇长篇科普摘抄 | 运输跌落 → 永远片札 |

**与 in-context learning 同构：** 正文 = 给例子让读者拼；附录 = 显式 labels；重读 = 第二轮 in-context。

## 十二、2024–2026 市场结构（扩充调研摘要）

> 完整条目、链接与 Agent 调研清单见 **`references/contemporary-structures-and-markets.md`**。写稿前若用户要「更火」「更像爆款」，先读该文件区分赛道。

### 12.1 三条赛道，不要混验收

1. **流量型**（网文/盐言/短剧）：密钩子、密爽点、人设反差。  
2. **口碑型**（出版/BookTok）：双线、内省、体裁杂交、系统当反派。  
3. **知识叙事型**（本技能）：隐喻世界 + 信号系统 + 附录诚实。

知识长篇 **不要**整本套短剧「15 秒一转」；外篇可借 **强场景开篇 + 情感未闭合章末**。

### 12.2 可杂交的结构变体（在三线之外）

| 变体 | 适用场景 |
|------|----------|
| **双时间线** | 外篇「训练日/发布日」；内篇保持单线 |
| **循环结尾** | 主题含「单次 session 消亡、权重留存」 |
| **系统反派** | 外篇对抗算法/合规/数据流水线，非卡通恶人 |
| **类型骨架** | 用悬疑/哲学质问章扛峰值（如「中国房间」） |
| **内宇宙+外宇宙** | 2024 中文科幻讨论：科技想象与情感必须同时成立（[中国作家网·2024关键词](https://www.chinawriter.com.cn/n1/2025/0125/c404034-40409001.html)） |

### 12.3 当代出版侧趋势（知识作者可借）

- 文学fiction **借惊悚/科幻/恐怖的节奏**扛主题，而非加讲义（[Our Culture 2026](https://ourculturemag.com/2026/02/21/6-modern-literary-trends-shaping-contemporary-fiction/)）。  
- **内省回流**：句子承担情节功能；外篇宜细腻（[Tumbleweed 2026](https://tumbleweedwords.com/literary-fiction-2026-trends.html)）。  
- **AI 题材升温**：近未来 + 伦理两难 + 像当下新闻（[Draft2Book·AI fiction](https://draft2book.com/ten-exciting-fiction-storytelling-trends-in-early-2025/)）。  
- **Hopepunk / cli-fi / romantasy** 等与知识长篇关系弱；勿为追热点改统御隐喻。

## 十三、节奏与钩子（知识长篇专用）

| 元素 | 流量型常见值 | 知识长篇建议 |
|------|--------------|--------------|
| 情绪峰间隔 | ~1.8 章 | 外篇 2–4 章一小峰；内篇可按「一层塔/一个机制」 |
| 章末 | 威胁式揭秘钩 | 情感/伦理悬置；或暗格（非 cliffhanger） |
| 开篇 | 3000 字内危机+金手指 | 一场 **具体事故**（数据/伦理/关系） |
| 概念释放 | 悬念优先 | **认知渐显**；议会章类高潮放在机制「现身」时 |

**节拍自检：** 连续两章能否用一句话说清「读者情绪变化」？不能则加外篇戏或删重复寓言。

## 十四、声音与场景：反「观点库」

读者（含 Reddit/社区）对 **模板金句** 敏感。避免：

```
Rgb doesn't make it faster, it's purely taste.
```

改为 **一个画面 + 一个立场**：

```
I stuck to one accent color on my last build—full rainbow got old in a week.
```

**规则：**

- 每段至少一个可验证细节（物件、时间、身体感、具体失败）。  
- 英文：句首大写；`I`/`I'd`/`don't`/`it's` 规范；可 `lol/kinda`，勿满篇故意错拼。  
- 中文外篇：动作与对话先行；少用「第一…第二…」总结腔。

---

## 十五、AI 协作边界

可用 AI：灵感、设定网、冲突点罗列、章纲表。  
**人类必须保留：** 统御隐喻、外篇伦理抉择、终稿文风、信号层纪律（侧批禁技术词）。

禁止：AI 批量生成「本章知识点小结」或正文技术注释。

参考：[AI 赋能科幻创意写作课程](https://www.kpcswa.org.cn/xwzx/hydt/zppl/art/2026/art_245d5ba0888c444cb7c9b92d29552946.html)

## 十六、参考实现：《字之城》draft-v1

仓库：[aaab3/and · novel/draft-v1/novel](https://github.com/aaab3/and/tree/novel/draft-v1/novel)（21 章 + 暗格 + 隐喻崩塌表 + 进阶阅读）。

| 章节示例 | 结构/技巧示范 |
|----------|----------------|
| 01《半截》 | 内篇诞生 + 外篇 `grep "妈妈"` + 古篇 Shannon |
| 03《议会》 | Attention 寓言 + 热图外篇 + Vaswani 古篇 |
